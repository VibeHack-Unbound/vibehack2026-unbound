## Quick Start

This repo uses [mise](https://mise.jdx.dev/installing-mise.html) for runtime setup. `mise.toml` pins Node `26`, and `package.json` pins pnpm `11.1.2`.

Install mise and let its installer add shell activation for zsh:

```bash
curl https://mise.run/zsh | sh
```

If you use bash, fish, Windows, Homebrew, or another install path, use the [official mise install docs](https://mise.jdx.dev/installing-mise.html).

Open a new terminal, then install the project runtime and dependencies:

```bash
mise install

npm install --global corepack@latest
corepack enable pnpm

pnpm install
pnpm dev
```

Local dev URLs:

- Web app: <http://localhost:3000>
- API: <http://localhost:8787>
- API docs: <http://localhost:8787/docs>
- API docs (Swagger): <http://localhost:8787/swagger>

Wrangler is installed locally as a dev dependency when `pnpm install` runs. Use it through pnpm so the workspace version is used:

```bash
pnpm --filter unbound-api exec wrangler --version
pnpm --filter unbound-web exec wrangler --version
pnpm --filter unbound-api exec wrangler login
```

For a fresh package or to reinstall Wrangler in a package, Cloudflare's pnpm install command is:

```bash
pnpm add -D wrangler@latest
```

## Architecture

This is a two-app `pnpm` workspace deployed as two Cloudflare Workers:

- `web` renders the public application with TanStack Start, React, Vite, and server-side rendering.
- `api` exposes the JSON API with Hono, validates requests with Zod/OpenAPI schemas, and persists data through Drizzle ORM on Cloudflare D1.

The frontend and API are intentionally separated. The website should call public API endpoints over HTTP; it should not receive D1 bindings, database credentials, provider API keys, or other secrets. Database-backed and secret-backed server behavior belongs in `api`.

### Web app

Key files:

- `web/vite.config.ts`: Vite configuration. It wires together the Cloudflare Vite plugin, Tailwind CSS v4, TanStack Start, React, and the React Compiler Babel preset.
- `web/wrangler.jsonc`: Cloudflare Worker configuration for the SSR website. The Worker entrypoint is `@tanstack/react-start/server-entry`.
- `web/src/router.tsx`: Creates the TanStack Router instance from the generated route tree.
- `web/src/routeTree.gen.ts`: Generated TanStack Router route tree. Do not edit this file by hand.
- `web/src/routes/__root.tsx`: Root document shell. It defines `<html>`, `<head>`, global metadata, stylesheet loading, and TanStack scripts.
- `web/src/routes/index.tsx`: The `/` route. At the moment this is a placeholder route that renders no UI.
- `web/src/styles.css`: Global CSS. Tailwind v4 is enabled through `@import 'tailwindcss';`; there is no separate Tailwind config file in the current tree.

TanStack Start provides SSR for the React app on Cloudflare Workers. Routes live under `web/src/routes`, and the generated route tree keeps the router type-safe. When adding pages, add route files under `web/src/routes` and let TanStack regenerate `routeTree.gen.ts`.

Tailwind is the styling system. Prefer Tailwind utility classes and shared CSS in `web/src/styles.css` before introducing a new styling abstraction. If adding component helpers, keep them in `web/src` and keep global design tokens or base styles in `styles.css` so both humans and AI agents can find them quickly.

### API app

Key files:

- `api/src/index.ts`: Hono Worker entrypoint. It configures CORS, OpenAPI route definitions, request validation, submission CRUD routes, docs endpoints, 404 handling, and error handling.
- `api/src/db/client.ts`: Creates a Drizzle client from the Cloudflare D1 `DB` binding.
- `api/src/db/schema.ts`: Drizzle schema for the `submissions` table and TypeScript row types.
- `api/migrations/`: D1 SQL migrations generated from the Drizzle schema.
- `api/drizzle.config.ts`: Drizzle Kit configuration for SQLite-compatible D1 migrations.
- `api/wrangler.jsonc`: Cloudflare Worker configuration for the API, including the `DB` D1 binding for production and staging.

The API uses `OpenAPIHono` plus `@hono/zod-openapi`, so route schemas are both runtime validators and OpenAPI documentation sources. The generated docs are available from the API Worker at `/docs`, `/swagger`, and `/openapi.json`.

The current domain model is a `submissions` table with `id`, `teamName`, `projectName`, optional `description`, optional `repoUrl`, optional `demoUrl`, and `createdAt`. Drizzle maps the TypeScript field names to SQLite column names such as `team_name` and `created_at`.

### Request flow

1. A browser requests the website Worker.
2. TanStack Start server-renders the matching route on Cloudflare Workers.
3. Browser-side React hydrates the route and handles client navigation.
4. When future UI needs backend data, it should call the API Worker over HTTP.
5. The API Worker validates the request, creates a Drizzle client from `c.env.DB`, and reads or writes Cloudflare D1.

### Deployment flow

Root scripts orchestrate both packages:

- `pnpm dev`: runs the web and API dev servers in parallel.
- `pnpm build`: builds the web app and typechecks the API.
- `pnpm typecheck`: typechecks both packages.
- `pnpm deploy:staging`: migrates the staging D1 database, deploys the staging API Worker, then deploys the staging web Worker.
- `pnpm deploy:production`: migrates the production D1 database, deploys the production API Worker, then deploys the production web Worker.

The API deploy scripts run D1 migrations before publishing the API Worker. If the schema changes, update `api/src/db/schema.ts`, generate a migration with `pnpm --filter unbound-api db:generate`, review the SQL in `api/migrations/`, then deploy.

### Boundaries for contributors and AI agents

- Keep frontend UI, routes, and styling in `web/src`.
- Keep API routes, validation schemas, and database access in `api/src`.
- Keep database schema changes in `api/src/db/schema.ts` and migrations in `api/migrations`.
- Do not edit `web/src/routeTree.gen.ts` manually.
- Do not add D1 bindings or secrets to `web/wrangler.jsonc`.
- Do not query D1 from the web app. Add or update API endpoints instead.
- Use Tailwind v4 utilities for styling; add global CSS only when utility classes are not enough.
- Prefer updating existing Hono OpenAPI routes over adding undocumented API handlers.

## Cloudflare Setup

The repo deploys two Workers:

- `unbound`: TanStack Start SSR website.
- `unbound-api`: Hono API Worker backed by D1.

Before deploying, replace `your-workers-subdomain` in these files with the Workers subdomain for your Cloudflare account:

- `web/.env.production`
- `web/.env.staging`

The free `workers.dev` URLs are:

- Production web: `https://unbound.<your-workers-subdomain>.workers.dev`
- Staging web: `https://unbound-staging.<your-workers-subdomain>.workers.dev`
- Production API: `https://unbound-api.<your-workers-subdomain>.workers.dev`
- Staging API: `https://unbound-api-staging.<your-workers-subdomain>.workers.dev`

Initialize Cloudflare resources:

```bash
pnpm install
pnpm --filter unbound-api exec wrangler login

pnpm --filter unbound-api exec wrangler d1 create unbound-api
pnpm --filter unbound-api exec wrangler d1 create unbound-api-staging
```

Paste the returned D1 IDs into `api/wrangler.jsonc` under `env.production.d1_databases[0].database_id` and `env.staging.d1_databases[0].database_id`.

Deploy. The API deploy script runs the matching D1 migrations before publishing the API Worker:

```bash
pnpm deploy:staging

pnpm deploy:production
```

Targeted deploys:

```bash
pnpm deploy:api:staging
pnpm deploy:web:staging
pnpm deploy:api:production
pnpm deploy:web:production
```
